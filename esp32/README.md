# ESP32-S3 firmware (scaffold)

On-device ESC-50 classifier for the ESP32-S3 (N16R8). This is the **flash-ready
structure** written before the board was in hand — the inference graph, model
embedding, quantization I/O, config, and dashboard bridge are done; two pieces
need the board + a clean model to finish (below).

## Layout
- `platformio.ini` — board = `esp32-s3-devkitc-1`, PSRAM on, TFLite Micro +
  esp-dsp + WebSockets deps.
- `src/config.h` — **all audio/model params in one place**, must match training
  (`scripts/common.py`): SR 22050, 64 mels, 5s, 216 frames. Also I2S pins + WiFi.
- `src/main.cpp` — TFLite Micro setup, int8 quantize using the tensor's own
  scale/zero-point, top-1 decode, latency print. Runs on a zero input during
  bring-up so you can confirm the model loads before wiring the mic.
- `src/mel_frontend.{h,cc}` — on-device log-mel front end (framing + normalization
  done; **FFT/mel kernels stubbed**).
- `src/esc50_model/model_data.{cc,h}` — the model as a C byte array, generated by
  `scripts/export_esp32.py` (`.cc` is gitignored — regenerate it).

## Generate the model array
```bash
../esc50env/bin/python3 ../scripts/export_esp32.py --tflite <clean_int8>.tflite
```

## Build / flash (once the board is here)
```bash
pio run                 # build
pio run -t upload       # flash
pio device monitor -b 115200
```

## Two things that still need doing
1. **A trustworthy fully-INT8 `.tflite`.** The local TF→TFLite path hangs
   (DECISIONS.md M4); the existing `tflite_models/mid_student*` files are from
   that flaky path and their sizes look wrong. Regenerate the QAT students'
   int8 tflite in a fresh env or Colab, then export.
2. **`mel_frontend.cc` FFT/mel math.** Fill the two stubbed kernels with esp-dsp,
   then A/B-check a few frames on-device against `common.wav_to_mel` on the same
   wav. A mel front end that is subtly wrong = great val accuracy, garbage
   on-device. Precompute the 64 triangular mel filters on the host and ship them
   as a const table rather than building them on the MCU.

Everything else (WiFi bridge to `dashboard/server.py::broadcast`, live latency
reporting) is wired and just needs the above two to be real.
